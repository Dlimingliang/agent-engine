## Python Agent 学习路线图（3 个月）

### 目标与原则

- **技术栈固定**：全程使用 Python。
- **路线顺序**：先单 Agent 基础能力，再工作流与多轮会话，再长期记忆与知识系统，再工具协议与技能化，最后才进入多 Agent 与 A2A。
- **核心原则**：先理解运行时与状态，再上框架；先把单 Agent 做稳，再考虑 Agent 之间协作。
- **最终目标**：你能独立设计并实现一个具备业界常见属性的 Agent 系统，而不是只会调某个框架 API。

---

## 第一阶段：打稳单 Agent 基础（第 1-3 周）

### 第 1 周：搭建最小 Agent Runtime

- **学习重点**
  - LLM 调用抽象
  - 消息模型（system / developer / user / assistant / tool）
  - 工具调用基础
  - 结构化输出
  - 观察-思考-行动-反馈闭环

- **必须掌握的属性**
  - **Prompt 分层**：不要把所有提示词写成一坨字符串。
  - **Tool Calling**：工具 schema、参数校验、工具结果回填。
  - **基础状态**：每一步执行日志、工具调用记录、模型输入输出记录。

- **实践项目**
  - 做一个命令行 Agent：
    - 能回答问题
    - 能调用 2-3 个本地工具
    - 能输出结构化结果
    - 能记录每轮执行轨迹

- **产出要求**
  - 一个 `Agent` 类
  - 一个 `ToolRegistry`
  - 一个 `Message` 数据结构
  - 一份最小 trace 日志

- **不要做的事**
  - 先不要上多 Agent
  - 先不要碰 A2A
  - 先不要一上来接复杂框架

### 第 2 周：Prompt、Rule、Guardrail 体系化

- **学习重点**
  - system prompt、task prompt、tool prompt、output contract 分层
  - rule 注入与优先级管理
  - 安全约束、行为边界、拒答策略
  - 输出格式控制

- **必须掌握的属性**
  - **Rule 系统**：静态规则、动态规则、用户规则、工具规则分开管理。
  - **冲突处理**：规则冲突时谁优先。
  - **输出契约**：JSON schema / pydantic model 约束。

- **实践项目**
  - 扩展你的 Agent：
    - 支持注入不同规则集
    - 支持输出格式模板
    - 支持拒答策略
    - 支持“不同任务使用不同 prompt 片段”

- **你要回答的问题**
  - prompt 是字符串技巧，还是配置系统？
  - rule 应该在运行前注入，还是运行时动态拼装？

### 第 3 周：多轮对话与会话管理

- **学习重点**
  - session 概念
  - conversation history 管理
  - 上下文裁剪与摘要
  - 轮次状态追踪
  - 中断后恢复会话

- **必须掌握的属性**
  - **会话管理**：会话 ID、用户 ID、消息列表、消息元数据。
  - **多轮对话**：历史消息如何保留，如何压缩。
  - **上下文窗口控制**：何时截断，何时摘要。
  - **回合状态**：当前任务是否完成、是否等待用户、是否需要工具结果。

- **实践项目**
  - 把你的 Agent 升级成一个多轮聊天 Agent：
    - 支持多个独立会话
    - 支持会话历史持久化
    - 支持历史压缩摘要
    - 支持重启后恢复最近会话

- **本阶段里程碑**
  - 你已经不是在做“单次问答机器人”，而是在做真正的 **Conversation Agent**。

---

## 第二阶段：记忆与知识供给（第 4-6 周）

### 第 4 周：短期记忆、长期记忆与工作记忆

- **学习重点**
  - short-term memory
  - long-term memory
  - working memory / scratchpad
  - episodic memory 与 semantic memory 区分

- **必须掌握的属性**
  - **短期记忆**：当前会话上下文、最近轮次关键点。
  - **长期记忆**：用户偏好、稳定事实、持久决策。
  - **工作记忆**：当前任务的中间计划、已完成步骤、待处理事项。
  - **记忆写入策略**：不是每条信息都存。
  - **记忆检索策略**：什么时候查历史，什么时候不查。

- **实践项目**
  - 给 Agent 增加记忆模块：
    - 自动识别用户偏好
    - 存储稳定信息
    - 查询历史偏好并在后续回答中生效
    - 区分“临时状态”和“长期记忆”

- **关键理解**
  - 记忆不是把所有聊天记录塞回模型。
  - 真正的记忆系统要解决 **存什么、何时存、怎么取、取多少**。

### 第 5 周：RAG 与知识库系统

- **学习重点**
  - 文档切片（chunking）
  - embedding
  - 向量检索
  - hybrid search
  - rerank
  - 引用与溯源

- **必须掌握的属性**
  - **知识库接入**：文档导入、索引构建、检索接口。
  - **检索质量**：召回率、相关性、去重。
  - **上下文拼装**：检索结果如何进入 prompt。
  - **引用链路**：回答引用哪些片段。

- **实践项目**
  - 做一个知识问答 Agent：
    - 导入你自己的技术文档
    - 支持检索增强回答
    - 支持返回引用来源
    - 支持检索失败时降级回答

- **不要误解**
  - RAG 不是 Agent 的全部。
  - RAG 是“知识供给模块”，不是 Agent runtime 本身。

### 第 6 周：记忆 + RAG 融合

- **学习重点**
  - 个人记忆与外部知识的边界
  - memory retrieval 与 document retrieval 的协同
  - 多源上下文合并
  - context packing 策略

- **必须掌握的属性**
  - **双检索**：先查用户记忆，再查知识库。
  - **上下文融合**：用户偏好、当前会话、外部文档如何一起进模型。
  - **优先级控制**：历史记忆不能污染事实性知识。

- **实践项目**
  - 做一个“懂用户 + 懂文档”的 Agent：
    - 能记住用户偏好
    - 能用文档回答问题
    - 回答时能区分“这是用户偏好”还是“这是文档事实”

- **本阶段里程碑**
  - 你已经掌握业界 Agent 最关键的两类上下文来源：**记忆** 与 **知识**。

---

## 第三阶段：工作流、可控执行与评估（第 7-8 周）

### 第 7 周：Planner、Executor、Verifier 单体化设计

- **学习重点**
  - plan-and-execute
  - self-critique / verifier
  - 失败重试
  - 分步执行
  - 工具错误恢复

- **必须掌握的属性**
  - **计划能力**：把复杂任务分成步骤。
  - **执行能力**：逐步执行并更新状态。
  - **验证能力**：检查结果是否满足目标。
  - **重试机制**：失败后重新规划还是重试当前步。

- **实践项目**
  - 把你的 Agent 做成一个可执行任务的助手：
    - 先规划
    - 再执行
    - 最后校验
    - 出错后有重试或回退

- **关键理解**
  - 这时仍然是 **单 Agent 内部分层**，不是多 Agent。
  - 不要过早把 planner 和 executor 拆成多个 agent 进程。

### 第 8 周：可观测性、评估与回放

- **学习重点**
  - tracing
  - step logging
  - prompt/version 管理
  - replay
  - offline eval

- **必须掌握的属性**
  - **可观测性**：每次运行看得到 prompt、工具调用、上下文、结果。
  - **评估**：回答正确率、工具调用成功率、检索命中率。
  - **回放**：能复盘某次失败执行。
  - **版本管理**：prompt 版本、rule 版本、tool schema 版本。

- **实践项目**
  - 为你的 Agent 增加调试台账：
    - 查看每轮输入输出
    - 查看工具调用链
    - 查看记忆读取记录
    - 查看检索来源

- **本阶段里程碑**
  - 你开始从“能跑起来”进化到“能维护、能调优、能定位问题”。

---

## 第四阶段：框架化与协议化（第 9-10 周）

### 第 9 周：引入主框架，但只让它服务于你的设计

- **推荐优先级**
  - **第一优先**：`LangGraph`
  - **第二优先**：`PydanticAI`

- **学习重点**
  - 状态图
  - 节点编排
  - 中断恢复
  - human-in-the-loop
  - typed agent I/O

- **必须掌握的属性**
  - **状态流**：节点之间传的到底是什么状态。
  - **图式执行**：何时进入哪个节点，何时结束。
  - **人工介入**：需要确认时如何挂起。
  - **结构化上下文**：状态对象而不是散乱字符串。

- **实践项目**
  - 用框架重构你之前的 Agent：
    - 保留你自己的消息、记忆、工具抽象
    - 把执行流迁移到图结构中
    - 支持中断与恢复

- **学习要求**
  - 你要始终知道：框架替你做了什么，你自己保留了什么。

### 第 10 周：MCP 与技能化（skills）

- **学习重点**
  - MCP 基本协议
  - MCP server / client
  - tool discovery
  - 工具 schema 标准化
  - skills 的能力封装思路

- **必须掌握的属性**
  - **MCP**：把工具能力协议化，而不是写死在代码里。
  - **Skills**：把高层能力模块化，形成“能力包”。
  - **能力路由**：什么时候调用哪个 skill 或 MCP 工具。
  - **权限控制**：哪些工具可用、哪些受限。

- **实践项目**
  - 给 Agent 接一个 MCP server
  - 把一组常用操作封装成 skill
  - 让 Agent 能按任务自动选择工具或 skill

- **关键理解**
  - MCP 解决的是 **能力接入标准化**。
  - skill 解决的是 **能力组织和复用**。

---

## 第五阶段：接近业界 Agent 系统（第 11 周）

### 第 11 周：完整 Agent 应用组装

- **这一周的目标**
  - 组装一个业界常见属性较完整的 Agent 系统。

- **系统必须包含**
  - 多轮对话
  - 会话管理
  - 短期记忆
  - 长期记忆
  - RAG 检索
  - rule / prompt 分层
  - 工具调用
  - MCP 接入
  - skill 机制
  - 计划与执行
  - 结果验证
  - 失败重试
  - trace / replay

- **实践项目建议**
  - 做一个“个人技术助理 Agent”或“项目 Copilot Agent”：
    - 能记住用户长期偏好
    - 能读取知识库
    - 能执行工具
    - 能管理多轮会话
    - 能记录执行链路
    - 能通过 MCP 接入外部能力

- **验收标准**
  - 不是看功能多，而是看系统边界是否清晰。

---

## 第六阶段：最后才进入多 Agent 与 A2A（第 12 周及之后）

### 第 12 周：多 Agent 之前，先回答这几个问题

在进入 A2A 之前，你必须先搞清楚：

- 单 Agent + 多工具，为何不够？
- planner、executor、reviewer 是否真的需要拆分？
- 多个 agent 之间传递的是任务、状态、上下文还是结果？
- 失败时如何重试、回滚与仲裁？
- 权限与信任边界如何管理？

### 第 13 周：多 Agent 基础协作

- **学习重点**
  - 角色分工
  - 委托模式
  - 共享状态 vs 消息通信
  - 中央调度 vs 去中心化协作

- **实践项目**
  - 做一个 2-3 个角色的小型系统：
    - planner
    - executor
    - reviewer

- **关键要求**
  - 先在单进程内实现协作逻辑。
  - 暂时不要一上来搞分布式。

### 第 14 周：A2A 入门（最后阶段）

- **学习重点**
  - capability discovery
  - task delegation
  - agent identity
  - task lifecycle
  - 跨 agent 协议交互

- **这时你才开始关注**
  - A2A 协议设计
  - 不同 agent 的互操作
  - 远程能力发现
  - 多 agent 网络中的状态流转

- **实践项目**
  - 让一个主 Agent 把某类任务委托给另一个 Agent
  - 能追踪任务状态
  - 能接收子 Agent 结果并汇总

- **最终里程碑**
  - 你会明白：A2A 不是“更高级的聊天”，而是 **跨 Agent 系统协作问题**。

## 推荐学习顺序总结

### 必修能力顺序

1. **单 Agent runtime**
2. **prompt 与 rule 分层**
3. **多轮对话与会话管理**
4. **短期记忆 / 长期记忆 / 工作记忆**
5. **RAG 与知识供给**
6. **计划、执行、验证、重试**
7. **可观测性与评估**
8. **框架化编排（LangGraph / PydanticAI）**
9. **MCP 与 skills**
10. **完整 Agent 系统集成**
11. **多 Agent**
12. **A2A（最后）**

### 推荐框架顺序

- **第一个重点学**：`LangGraph`
- **第二个补充学**：`PydanticAI`
- **RAG 相关**：可按需了解 `LlamaIndex`，但不要让它主导你的 Agent runtime 设计

---

## 学习中的常见误区

- **误区 1**：把 RAG 当作 Agent 本体
- **误区 2**：把会话历史当作记忆系统
- **误区 3**：一上来就做多 Agent
- **误区 4**：一开始就沉迷框架 API，不理解底层运行时
- **误区 5**：没有 trace、评估、回放，就急着追求复杂能力
- **误区 6**：还没把单 Agent 做稳，就提前做 A2A

---

## 你接下来最值得做的第一件事

**不要先学 A2A。**
先在 Python 里亲手做一个具备以下能力的单 Agent：

- 多轮对话
- 会话管理
- 规则注入
- 工具调用
- 短期记忆
- 长期记忆
- 简单 RAG
- trace 日志

当你把这一个系统做扎实后，再引入 `LangGraph` 做状态编排，再继续走向 MCP、skills、多 Agent，最后才是 A2A。

---

## 最后一句路线建议

**你的目标不是“学会几个 Agent 名词”，而是建立一套可以稳定演进的 Python Agent 系统观。**

这条路线的核心节奏只有一句话：

**先单体、后工作流；先上下文、后协作；先系统能力、后 A2A。**

## Python Agent 按周学习清单 + 每周项目题目

### 使用说明

- **周期**：14 周。
- **节奏**：每周 5-7 小时学习 + 5-8 小时动手。
- **输出原则**：每周必须有可运行产物，不接受只看不做。
- **路线原则**：先单 Agent，后工作流；先会话与记忆，后多 Agent；**A2A 永远放最后**。

---

## 第 1 周：最小 Agent Runtime

### 本周目标

把 Agent 最基础的执行闭环跑通：输入问题、组织消息、调用模型、必要时调用工具、输出结果。

### 学习清单

- **理解消息模型**：`system`、`user`、`assistant`、`tool` 分别承担什么职责。
- **理解 Agent 最小闭环**：输入 -> 思考/决策 -> 工具调用 -> 结果回填 -> 输出。
- **理解工具 schema**：工具为什么要有名称、描述、参数结构。
- **理解结构化输出**：为什么很多 Agent 不能只靠自然语言返回。
- **理解 trace 基础**：每一步为什么必须可记录、可复盘。

### 本周项目题目

- **题目**：做一个命令行版 `Mini Agent`
- **要求**：
  - 接收用户问题
  - 支持 2-3 个工具（如时间查询、文件读取、网页获取中的模拟版）
  - 支持一次工具调用闭环
  - 输出最终答案
  - 把执行过程记录到日志文件

### 本周交付物

- `Agent` 主类
- `Message` 数据结构
- `ToolRegistry`
- 一份可读的 trace 日志

### 验收标准

- 能完成至少 3 类不同任务
- 工具调用失败时不会直接崩溃
- 日志中看得出每一步发生了什么

---

## 第 2 周：Prompt 分层与 Rule 系统

### 本周目标

把“提示词技巧”升级为“提示词配置系统”，让 Agent 行为可控、可复用、可约束。

### 学习清单

- **拆 Prompt 层次**：system prompt、task prompt、tool prompt、output contract。
- **理解 Rule 分类**：安全规则、产品规则、用户偏好规则、工具规则。
- **理解优先级**：冲突时什么规则最高。
- **理解拒答与降级**：何时拒绝、何时改写、何时请求澄清。
- **理解结构化约束**：用 `pydantic` 或 schema 限制输出。

### 本周项目题目

- **题目**：给 `Mini Agent` 加上 `PromptComposer + RuleEngine`
- **要求**：
  - 按任务类型切换 prompt 片段
  - 支持静态规则与动态规则注入
  - 支持结构化输出模式
  - 支持 2 种拒答策略

### 本周交付物

- `PromptComposer`
- `RuleEngine`
- 至少 3 套可切换规则集
- 至少 2 个结构化输出模型

### 验收标准

- 相同问题在不同规则集下能产生不同但可预期的行为
- 输出格式稳定，不靠字符串硬解析

---

## 第 3 周：多轮对话与会话管理

### 本周目标

让 Agent 从“单次调用”变成“真正会聊天的系统”。

### 学习清单

- **理解 Session**：为什么要有 `session_id`、`user_id`。
- **理解 History 管理**：全部保留、截断、摘要三种方式的差别。
- **理解轮次状态**：当前回合完成、挂起、等待工具、等待用户。
- **理解上下文窗口**：什么时候该压缩历史。
- **理解会话恢复**：系统重启后如何接上上次对话。

### 本周项目题目

- **题目**：做一个 `Conversation Agent`
- **要求**：
  - 支持多会话并存
  - 每个会话可单独保存历史
  - 支持会话摘要
  - 支持会话恢复
  - 支持标记当前会话状态

### 本周交付物

- `SessionStore`
- `ConversationManager`
- 历史压缩摘要模块
- 会话恢复脚本

### 验收标准

- 至少可并行管理 3 个会话
- 不同会话不会串上下文
- 历史很长时仍能继续对话

---

## 第 4 周：短期记忆、长期记忆、工作记忆

### 本周目标

学会把“聊天记录”与“记忆系统”分开。

### 学习清单

- **短期记忆**：最近几轮的关键状态。
- **长期记忆**：用户偏好、长期事实、稳定决策。
- **工作记忆**：当前任务的计划、已完成步骤、待办。
- **理解写入策略**：什么该记，什么不该记。
- **理解读取策略**：什么时候注入记忆最合适。

### 本周项目题目

- **题目**：实现 `MemoryManager`
- **要求**：
  - 能保存用户长期偏好
  - 能保存当前任务的工作状态
  - 能按需回忆，而不是每次全量塞回模型
  - 能区分临时信息与长期信息

### 本周交付物

- `ShortTermMemory`
- `LongTermMemory`
- `WorkingMemory`
- 记忆写入与检索策略模块

### 验收标准

- Agent 能记住用户 3 条长期偏好并在后续回答中生效
- 当前任务中断后可恢复工作上下文

---

## 第 5 周：RAG 基础与知识库接入

### 本周目标

让 Agent 不只依赖参数知识，而是会从外部知识源取资料。

### 学习清单

- **理解 Chunking**：按段、按语义、按标题切片的差异。
- **理解 Embedding**：向量化的作用与局限。
- **理解向量检索**：召回不是答案。
- **理解 Rerank**：为什么仅向量召回经常不够。
- **理解引用**：答案为什么必须能溯源。

### 本周项目题目

- **题目**：做一个 `Doc QA Agent`
- **要求**：
  - 导入 20-50 篇文档或技术笔记
  - 构建检索索引
  - 支持检索增强回答
  - 支持返回引用来源
  - 检索不到时能降级回答

### 本周交付物

- 文档导入脚本
- 检索模块
- 引用输出模块
- 一个小型知识库样本

### 验收标准

- 至少 10 个问题中有 7 个能命中正确文档片段
- 回答能标出引用来源

---

## 第 6 周：记忆与 RAG 融合

### 本周目标

让 Agent 既“懂用户”，又“懂资料”，并能区分两者来源。

### 学习清单

- **理解双检索**：memory retrieval 与 document retrieval 的差异。
- **理解上下文融合**：记忆、会话、知识如何合并。
- **理解优先级**：用户偏好不能覆盖客观事实。
- **理解污染问题**：错误记忆如何伤害 RAG 回答。
- **理解上下文打包**：给模型多少上下文才合适。

### 本周项目题目

- **题目**：做一个 `Personal Knowledge Agent`
- **要求**：
  - 回答时能同时使用记忆与知识库
  - 明确区分“这是用户偏好”与“这是文档事实”
  - 在提示中显式标记上下文来源

### 本周交付物

- 双检索编排器
- context packing 模块
- 上下文来源标记规范

### 验收标准

- 至少 5 个测试问题中，回答能正确区分记忆来源与文档来源
- 不出现明显的“记忆污染事实”现象

---

## 第 7 周：计划、执行、验证、重试

### 本周目标

把 Agent 从“能聊”升级成“能做事”。

### 学习清单

- **理解 Plan-and-Execute**：为什么复杂任务要拆步骤。
- **理解 Verifier**：结果为什么要校验。
- **理解 Retry**：失败后是重试、改计划还是请求人工。
- **理解工具失败恢复**：错误信息如何进入下一步决策。
- **理解任务状态机**：任务进行中、已完成、失败、待确认。

### 本周项目题目

- **题目**：做一个 `Task Agent`
- **要求**：
  - 先生成执行计划
  - 分步执行
  - 每步后更新状态
  - 最后做结果验证
  - 出错后支持重试或改计划

### 本周交付物

- `Planner`
- `Executor`
- `Verifier`
- 任务状态跟踪模块

### 验收标准

- 至少能稳定完成 3 个多步骤任务
- 工具失败后能继续而不是直接结束

---

## 第 8 周：可观测性、调试、评估、回放

### 本周目标

让 Agent 系统进入“可维护状态”。

### 学习清单

- **理解 Trace**：如何记录一次执行的完整链路。
- **理解 Replay**：为什么失败案例要能回放。
- **理解 Eval**：如何评估回答质量、检索质量、工具质量。
- **理解版本管理**：prompt、rule、tool schema 为什么都要版本化。
- **理解 Debug 面板**：开发 Agent 时最想看到哪些信息。

### 本周项目题目

- **题目**：做一个 `Agent Trace Console`
- **要求**：
  - 能查看每轮输入输出
  - 能查看工具调用链
  - 能查看读取了哪些记忆和文档
  - 能回放一条失败任务
  - 能记录 prompt/rule 版本

### 本周交付物

- trace 存储结构
- replay 脚本
- 简易评估集
- 调试界面或命令行报表

### 验收标准

- 任何一次失败调用都能追出原因
- 至少能对 20 条样本做离线评估

---

## 第 9 周：LangGraph 入门与状态图编排

### 本周目标

开始用框架，但框架只负责“编排”，不接管你的全部设计。

### 学习清单

- **理解 State Graph**：节点之间共享什么状态。
- **理解 Node**：每个节点只负责什么单一职责。
- **理解条件分支**：不同状态如何走不同路径。
- **理解中断恢复**：图执行中途如何暂停。
- **理解 Human-in-the-loop**：何时需要人工确认。

### 本周项目题目

- **题目**：把 `Task Agent` 重构成 `LangGraph Agent`
- **要求**：
  - 用图表达 planner / executor / verifier 流程
  - 支持条件分支
  - 支持中断恢复
  - 保留你自己的消息、记忆、工具抽象

### 本周交付物

- 一个最小 LangGraph 状态图
- 状态对象定义
- 中断恢复示例

### 验收标准

- 你能清楚解释：框架做了什么，你自己仍控制什么
- 图结构比 if/else 版更清晰

---

## 第 10 周：PydanticAI 与结构化 Agent

### 本周目标

加强类型化、结构化、可验证的 Agent 编程方式。

### 学习清单

- **理解类型约束**：为什么 Agent 输入输出需要强约束。
- **理解结构化结果**：避免过多依赖自然语言解析。
- **理解工具参数校验**：先校验，再调用。
- **理解错误建模**：失败结果也要结构化。
- **理解 typed context**：上下文对象应该有明确字段。

### 本周项目题目

- **题目**：做一个 `Typed Agent`
- **要求**：
  - 所有工具输入输出都类型化
  - 关键任务结果结构化
  - 错误返回也结构化
  - 对外暴露统一接口

### 本周交付物

- `pydantic` 模型集合
- 类型化工具层
- 类型化任务结果层

### 验收标准

- 不再依赖脆弱的字符串匹配来驱动系统逻辑
- 错误处理路径清晰可测

---

## 第 11 周：MCP 与技能化（Skills）

### 本周目标

把 Agent 的能力从“代码里写死”升级为“可插拔、可协议化、可复用”。

### 学习清单

- **理解 MCP**：为什么工具能力需要标准协议。
- **理解 MCP Client/Server**：谁暴露能力，谁消费能力。
- **理解 Tool Discovery**：Agent 怎么知道有哪些能力可用。
- **理解 Skill**：高层能力包与单一工具的差别。
- **理解权限控制**：不同上下文下工具可见性为何不同。

### 本周项目题目

- **题目**：做一个 `MCP + Skills Agent`
- **要求**：
  - 接一个 MCP Server
  - 封装 2-3 个技能包
  - 按任务类型自动路由到不同能力
  - 给高风险工具加权限限制

### 本周交付物

- 一个最小 MCP Client
- 一个技能路由层
- 工具权限配置

### 验收标准

- 新增能力时，不需要大改 Agent 主逻辑
- Agent 能根据任务选择 skill 或 MCP tool

---

## 第 12 周：完整 Agent 系统集成

### 本周目标

把前面所有模块组装成一个比较完整的业界风格 Agent 应用。

### 学习清单

- **统一系统边界**：哪些是 runtime，哪些是 memory，哪些是 retrieval，哪些是 protocol。
- **统一状态流**：会话状态、任务状态、记忆状态怎么互相配合。
- **统一配置**：prompt、rules、tools、memory、retrieval 都如何配置化。
- **统一调试**：所有关键链路都能观察。
- **统一验收**：建立一套最小测试题集。

### 本周项目题目

- **题目**：做一个 `Personal Copilot Agent`
- **系统必须包含**：
  - 多轮对话
  - 会话管理
  - 短期记忆
  - 长期记忆
  - RAG 检索
  - rule / prompt 分层
  - 工具调用
  - MCP 接入
  - skills
  - 计划执行验证
  - trace / replay

### 本周交付物

- 完整 Agent 应用
- 一份系统架构说明
- 一份测试题集
- 一份已知问题清单

### 验收标准

- 这是一个能持续迭代的系统，而不是临时 demo
- 新加模块不会破坏已有边界

---

## 第 13 周：多 Agent 基础协作（仍然不碰 A2A）

### 本周目标

先理解“为什么要多个 Agent”，再决定如何拆。

### 学习清单

- **理解角色分工**：planner、executor、reviewer、researcher 各自职责。
- **理解协作方式**：共享状态、消息传递、委托模式。
- **理解调度方式**：中央调度与分散协作的优劣。
- **理解失败仲裁**：多个 Agent 冲突时怎么办。
- **理解边界**：哪些任务其实单 Agent 足够。

### 本周项目题目

- **题目**：做一个 `3 角色协作系统`
- **要求**：
  - `planner` 负责拆任务
  - `executor` 负责执行
  - `reviewer` 负责审查
  - 先在单进程内通过共享状态或消息队列实现

### 本周交付物

- 多角色状态流设计
- 最小协作协议
- 失败仲裁策略

### 验收标准

- 你能回答“为什么这里要多 Agent，而不是单 Agent + 多工具”
- 角色拆分不是为了炫技，而是为了降低复杂度

---

## 第 14 周：A2A 入门（最后一周才开始）

### 本周目标

在你已经做过单 Agent、会话、记忆、RAG、MCP、skills、多 Agent 协作之后，才进入 A2A。

### 学习清单

- **理解 Agent Identity**：Agent 如何被识别。
- **理解 Capability Discovery**：主 Agent 如何发现别的 Agent 会什么。
- **理解 Task Delegation**：如何把任务委托出去。
- **理解 Task Lifecycle**：已创建、执行中、完成、失败、取消。
- **理解远程协作问题**：信任、权限、超时、回调、结果归并。

### 本周项目题目

- **题目**：做一个 `A2A Demo`
- **要求**：
  - 一个主 Agent
  - 一个子 Agent
  - 主 Agent 把某类任务委托给子 Agent
  - 能追踪任务状态
  - 能汇总结果
  - 能处理子 Agent 失败或超时

### 本周交付物

- 最小 A2A 通信协议草案
- 主从 Agent 委托示例
- 状态跟踪与超时处理示例

### 验收标准

- 你理解 A2A 是系统协作，不是聊天升级版
- 你能指出 A2A 与 MCP、skills、多 Agent 的边界区别

---

## 建议的每周固定动作

### 每周都做

- **写一份周总结**：记录这周学到的核心抽象。
- **写一份失败案例**：至少复盘 1 次失败调用。
- **做一组回归测试**：避免新能力破坏旧能力。
- **画一张结构图**：强迫自己把系统边界说清楚。

### 每两周做一次

- **清理 Prompt**：把临时提示词整理成正式配置。
- **清理 Memory**：删除无价值记忆，调整写入策略。
- **清理 Tools**：看哪些工具其实应升格为 skill。
- **清理评估集**：补充失败样本。

## 你当前最适合的开局动作

如果你今天就要开始，不要先读一堆框架文档。  
先直接做 **第 1 周项目：`Mini Agent`**，然后按下面顺序推进：

1. **先跑通最小闭环**
2. **再补 prompt/rule**
3. **再补会话和记忆**
4. **再补 RAG**
5. **再补计划执行与可观测性**
6. **再引入 LangGraph / PydanticAI**
7. **再做 MCP / skills**
8. **最后才做多 Agent / A2A**

---

## 最后一句提醒

**如果你把第 1-12 周做好，A2A 只是自然延伸；如果前面没做扎实，A2A 只会变成复杂度放大器。**
